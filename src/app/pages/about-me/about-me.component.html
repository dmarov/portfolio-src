Career

I've started my career as a backend developer in 2017. It was my first experience of commercial software development and that's when I got acquainted with PHP and MySQL. At the same time I took a grasp on frontend technologies such as Gulp, Webpack, Sass, Babel.

The time has come when native Javascript was not enough in order to keep implementing complex solutions in efficient way. That's when I made decision to learn Vue framework which payed out later.

Path lead to resolve more complex problems related to outdated and unmaintainable infrastructure. Being experienced with Node.js In order to resolve some of existed problems I've implemented microservices. There are some of them:

  • SMS sending gateway - REST API to send SMS notifications using GSM gateway
  • PDF generator - REST API to convert HTML page to PDF (useful for printing invoices, and not just that)
  • JWT token generator - REST API to generate JWT token pairs (useful for authorization)

These services were very useful for each new website.

Since each service required separate environnment it required a lot of administering efforts, which I was also responsible for. I already had a lot knowledge about FreeBSD operating system, so it was time to apply this knowledge in practice.

Then I decided to specialize in frontend and move to bigger projects. That's when I've learned Typescript and Angular which became part of my main technology stack till current day. Such aspects as Search Engine Optimization (SEO), implementing end-to-end tests, pixel perfect markup, cross browser development, accessibility - this is what I'm most focused on at work.

What I stand for

At present I realize that I've experienced a lot and can formulate what kind of developer I am:

  • I am huge advocate for code quality in long term products, because it allows to reduce future work as well for other developers and reduces amount of potential bugs
  • I am huge advocate for uniform code formatting, because it reduces distractions and helps to focus on application architecture
  • I consider that ability to ask correct questions is the most valuable skill a developer can have
  • I consider that every significant consideration should have valid argumentation in order to be legit
  • I consider that actions of software developer should be based on principles
  • I consider that refactoring should be regarded as a part of a task rather than a a separate process
  • I consider that developer should not be locked into narrow specialization, but have opportunities to learn adjacent domains from colleagues
  • I am huge advocate for fast code integration such as "trunk based flow", because it reduces bugs due to complicated conflicts for a coast of releasing incomplete features
  • I consider that fixing bugs should be topmost priority, because released feature is done when it's working, and it's working when it has no bugs
  • I consider that Agile principles for software development are worth following, mainly because it allows to provide fast feedback. Agile is not a solution for every problem and has disadvantages, but I think that advantages outweight disadvantages considerably
  • I consider that long term product should be developed using BDD (Behavior Driven Development) process. Writing test after the fact results in poor test quality and poor code design. TDD (Test Driven Development) is a good approach as well but it has gone completely wrong in a sense of what for developer should write a test. BDD is the best approach I've experienced so far. I don't think it is necessary to follow BDD principles in order to get good software design, but it's very handy to have extra verification for correctness of behavior
  • I think that naming is the hardest part of development process, as once was said "There are only two hard things in Computer Science: cache invalidation and naming things."
